# 常見錯誤與除錯

本頁整理前處理、PINN 訓練、二期微調與最佳化最常遇到的問題。

## 1. GPU OOM

常見訊息：

```text
OOM when allocating tensor
ResourceExhaustedError
```

原因通常是 Transformer attention 記憶體需求太大，與 `batch_size * num_heads * window_size^2` 有關。

處理順序：

1. 降低 `batch_size`：`512 -> 256 -> 128 -> 64`。
2. 若仍不足，降低 `num_transformer_layers`。
3. 再不足，降低 `d_model` 或 `ff_dim`。
4. 二期程式已使用 mixed precision，通常優先降 batch 即可。

## 2. XLA 或 mixed precision 錯誤

常見訊息：

```text
Operand is not divisible by the block size
XLA_GPU_JIT
```

二期程式已有：

```python
ENABLE_XLA = False
tf.config.optimizer.set_jit(ENABLE_XLA)
```

若仍出錯，先確認沒有手動把 XLA 開回 True。必要時也可暫時關閉 mixed precision：

```python
policy = mixed_precision.Policy("float32")
mixed_precision.set_global_policy(policy)
```

## 3. `KeyError: l_dat`

原因：Keras 訓練歷史中沒有記錄到自訂 loss key，通常和 `train_step` 回傳值或訓練中斷有關。

處理：

- 確認 `train_step` 最後有回傳 `{"loss": ..., "l_dat": ...}`。
- 若訓練很早中斷，先看前面的真正錯誤。
- 暫時不要依賴該 key 畫圖，先確認模型能完整跑完。

## 4. 找不到訓練資料 CSV

常見訊息：

```text
FileNotFoundError: Master_Training_Data_Continuous3.csv
FileNotFoundError: Phase2_Training_Data2.csv
```

處理：

- 確認目前工作目錄是程式所在資料夾。
- 一期 PINN 需要 `Master_Training_Data_Continuous3.csv`。
- 二期 PINN 與二期最佳化需要 `Phase2_Training_Data2.csv`。
- 若資料在別的資料夾，修改程式中的資料路徑。

## 5. 最佳化找不到 PINN 結果資料夾

常見訊息：

```text
找不到可用的 PINN 結果資料夾
```

最佳化程式會檢查 `pinn_report_path` 指到的資料夾是否含有：

```text
learned_T.npy
learned_C.npy
calibrated_inflow_sy.npy
```

二期最佳化通常還需要：

```text
background_h_7d.npy
accurate_pred_h_7d.npy
```

處理：

- 打開最佳化程式，修改 `opt_config["pinn_report_path"]`。
- 使用絕對路徑可避免相對路徑混淆。
- 確認 PINN 訓練是否完整跑完，沒有只產生部分圖表。

## 6. 二期 transfer 權重載入失敗

常見原因：

- `phase1_weights_path` 路徑錯誤。
- 一期與二期井位數不同，部分 layer shape 不相容。
- 權重檔不是此版本模型產生的。

處理：

- 確認 `phase1_weights_path` 指向 `pinn_model.weights.h5`。
- 保持 `allow_partial_transfer=True`，允許部分載入。
- 若完全無法載入，可先把 `phase1_weights_path` 設為空字串或改成可用的一期權重，讓二期接近從頭訓練。

## 7. 預測起點前 context 不足

常見訊息：

```text
PREDICT_START 前需要至少 window_size 筆資料
```

原因：自回歸預測需要 `PREDICT_START` 前 7 天資料作為初始 window。

處理：

- 將 `PREDICT_START` 往後移。
- 補足前 7 天資料。
- 不建議直接縮短 `window_size`，除非重新評估模型設定。

## 8. 水位整體偏移 1 到 2 公尺以上

可能原因：

- 預測起點水位與模型輸出基準不同。
- scaler fit 的資料期間不合理。
- 訓練資料中有跳點或錯誤補值。
- 二期 transfer 權重和二期場域差異太大。

處理：

- 二期先看 `start_level_bias_correction.csv`。
- 比較 `background_h_7d_raw.npy` 與 `background_h_7d.npy`。
- 檢查 `Prediction_vs_Actual_Obs.png` 是整體偏移還是趨勢也錯。
- 若趨勢正確但起點偏移，可保留 `START_LEVEL_CORRECTION=True`。
- 若趨勢也錯，回頭檢查資料欄位與訓練區間。

## 9. 抽水流量全部接近 0

常見原因是井號或欄位對不上。

檢查：

- `Qw01` / `QW01` 是否存在。
- `QW010` 是否被誤改成 `QW10`。
- `KWH010`、`X_010`、`QW010` 是否同時存在。
- `X_**` 是否幾乎全為 0。

若 `X_**` 全為 0，代表 `KWH.diff()` 沒有偵測到電表增量，需檢查電表欄位是否為累積值。

## 10. 最佳化不可行

常見原因：

- 目標水位太嚴格。
- 安全緩衝太大。
- 最少開井數、最短開機時間等限制互相衝突。
- PINN 背景水位或影響矩陣不合理。

處理：

- 一期放寬 `FIXED_TARGET_H` 或 `SAFETY_BUFFER`。
- 二期放寬 `TARGET_UPPER_TOLERANCE_M`。
- 暫時降低 `MIN_ACTIVE_WELLS`。
- 暫時降低 `MIN_UP_TIME`。
- 檢查 `Reverse_Verification_Plot.png` 是否合理。

## 11. 最佳化結果看起來節能但水位太低

這代表排程可能過抽。

二期處理：

- 提高 `OVER_PUMPING_PENALTY`。
- 提高 `TARGET_TRACKING_PENALTY`。
- 降低 `TARGET_LOWER_TOLERANCE_M`。
- 改用 `TARGET_MODE="fixed"` 設定更明確的控制水位。

一期處理：

- 調整 `FIXED_TARGET_H`。
- 增加安全緩衝。

## 12. 圖表中文字亂碼

Windows 通常可用：

```python
matplotlib.rc("font", family="Microsoft JhengHei")
```

二期最佳化會優先找：

```text
SourceHanSansTC-Regular.otf
```

若沒有此字型，請放到程式同資料夾，或改成系統已安裝的中文字型。

## 13. PowerShell 顯示中文亂碼

程式輸出或 Markdown 在終端看起來亂碼時，可先執行：

```powershell
chcp 65001
$env:PYTHONUTF8=1
```

若檔案本身是 UTF-8，GitHub 或 VS Code 通常仍能正常顯示。
